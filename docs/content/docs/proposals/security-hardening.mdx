## Goals

- The binary should be pre-compiled and signed by maintainers using a trusted code signing certificate. Users can verify the signature before installing. This prevents tampering with the binary and ensures users are running a trusted version.
- Each node, can be trusted or not trusted. To be a trusted node, the operator must use their crypto wallet to sign a message and endorse the node's identity. Users can choose to only route requests through trusted nodes. This creates a web of trust and allows users to avoid potentially malicious nodes. By default, even if the operator sign, the node is still not trusted until 1) the operator passed the KYC process (which can be done later by a centralized database) 2) the operator is particularly trusted by the user who send the requests. 3) the user trust themselves anyway (e.g., if they have the same wallet address).
- Implement rate limiting and request throttling to prevent abuse and DDoS attacks. Nodes can enforce limits on the number of requests per second from a single IP or wallet address.
- Use secure communication protocols (e.g., TLS) for all network traffic between nodes to prevent eavesdropping.

## Execution Plan

### Phase 1: Binary Signing (Low effort, high trust gain)

**Current state:** Release workflow (`.github/workflows/release.yml`) builds `otela-amd64` and `otela-arm64` binaries and publishes them to GitHub Releases unsigned.

**Steps:**

1. **Add cosign to the release workflow.** After building binaries, use [Sigstore cosign](https://github.com/sigstore/cosign) to sign each artifact. Cosign's keyless mode integrates with GitHub Actions OIDC — no secret key management needed.
   - Add step: `cosign sign-blob --yes --bundle otela-amd64.bundle otela-amd64`
   - Upload `.bundle` signature files alongside binaries in the GitHub Release.
2. **Add SHA256 checksums.** Generate a `checksums.txt` file (`sha256sum otela-*`) and sign it as well.
3. **Document verification.** Add a section to the README or install docs showing users how to verify:
   ```bash
   cosign verify-blob --bundle otela-amd64.bundle otela-amd64
   sha256sum -c checksums.txt
   ```

**Files to change:**
- `.github/workflows/release.yml` — add cosign sign + checksum steps
- `README.md` or install docs — verification instructions

### Phase 2: TLS for Node Traffic (Low effort, already partially done)

**Current state:** `host.go` already configures both TLS (`libp2ptls`) and Noise security transports. libp2p negotiates encryption automatically between peers, so **inter-node traffic is already encrypted**. However, the PSK (pre-shared key) code for network-level isolation is commented out (lines 106-109 in `host.go`).

**Steps:**

1. **Verify current encryption.** Add a log line or metric that confirms which security transport was negotiated per connection (TLS vs Noise). This provides observability.
2. **Consider enabling PSK for private networks.** Uncomment and expose the PSK config option so operators can optionally restrict their network to nodes sharing a secret key. Add a config field:
   ```yaml
   network:
     psk: "<base64-encoded-key>"  # optional, for private networks
   ```
3. **Enforce minimum TLS version.** Ensure TLS 1.3 is the minimum when TLS transport is used (this is the libp2p default, but worth an explicit assertion or test).

**Files to change:**
- `src/internal/protocol/host.go` — PSK config, logging
- Config schema / Viper bindings

**Note:** This phase may already be effectively complete. The main action is verification and observability.

### Phase 3: Rate Limiting (Medium effort)

**Current state:** The HTTP server (`server.go`) uses Gin with no rate limiting middleware. `ResourceManager` is set to `NullResourceManager` (no libp2p-level limits).

**Steps:**

1. **HTTP-level rate limiting.** Add a Gin middleware using a token-bucket or sliding-window algorithm. Key by client IP (from `c.ClientIP()`).
   - Config:
     ```yaml
     security:
       rate_limit:
         enabled: true
         requests_per_second: 100
         burst: 200
     ```
   - Use an in-memory store (e.g., `golang.org/x/time/rate` or `github.com/ulule/limiter`). No external dependency like Redis needed for single-node limiting.
   - Return `429 Too Many Requests` with `Retry-After` header when limit is hit.

2. **Per-wallet rate limiting (optional, layered on top).** If the request includes a wallet-identifying header (e.g., signed auth token), apply a separate per-wallet limit. This is a later refinement — start with IP-based.

3. **libp2p resource manager.** Replace `NullResourceManager` with `rcmgr.NewResourceManager()` using sensible defaults to bound per-peer connection/stream counts at the protocol level. This protects against P2P-layer abuse independently of HTTP.

**Files to change:**
- New file: `src/internal/server/ratelimit.go` — middleware implementation
- `src/internal/server/server.go` — register middleware
- `src/internal/protocol/host.go` — configure resource manager
- Config schema / Viper bindings

### Phase 4: Node Trust Model (High effort, phased rollout)

**Current state:** The `Peer` struct in `node_table.go` has a `Privileged` field (unused). Wallet signing exists in `internal/wallet/`. The node table is CRDT-replicated. Proxy routing in `proxy_handler.go` selects peers by identity group matching but has no trust filtering.

#### Phase 4a: Wallet-Signed Node Identity

1. **Node self-attestation.** On startup, the node signs a message containing `{peerID, walletPubkey, timestamp}` using its Ed25519 wallet key. This "identity attestation" is stored in the CRDT alongside the node's peer record.
   - Add fields to `Peer`: `WalletPubkey string`, `IdentityAttestation []byte`, `TrustLevel int`.
   - `TrustLevel` enum: `0 = untrusted`, `1 = self-attested`, `2 = user-trusted`, `3 = KYC-verified`.

2. **Attestation verification.** When a node receives a peer record via CRDT/gossip, it verifies the Ed25519 signature. Invalid attestations are rejected (peer treated as untrusted).

**Files to change:**
- `src/internal/protocol/node_table.go` — new fields, attestation verification
- `src/internal/wallet/wallet.go` — `SignAttestation()` / `VerifyAttestation()` helpers
- `src/internal/protocol/registrar.go` — sign attestation on registration

#### Phase 4b: Trust-Aware Routing

3. **Request header for trust preference.** Clients send `X-Otela-Trust: <min-level>` (0-3). Default: `0` (accept any node).
4. **Filter in proxy routing.** In `proxy_handler.go`, after identity-group matching, filter candidates by `TrustLevel >= requested level`. If no trusted peers are available, return `503` with a descriptive error (don't silently fall back to untrusted).

**Files to change:**
- `src/internal/server/proxy_handler.go` — trust filtering logic
- `src/internal/server/cors.go` — allow `X-Otela-Trust` header

#### Phase 4c: Per-User Trust Lists

5. **User trust store.** Users can explicitly trust specific wallet addresses. Stored locally in the user's config:
   ```yaml
   trusted_wallets:
     - "5YourFriend..."
     - "7AnotherOperator..."
   ```
6. **Self-trust shortcut.** If the requesting user's wallet matches the node's wallet, auto-elevate to `TrustLevel = 2`.

**Files to change:**
- Config schema
- `src/internal/server/proxy_handler.go` — per-user trust resolution

#### Phase 4d: KYC Integration (Future)

7. **KYC oracle.** A centralized service maintains a list of KYC-verified wallet addresses. Nodes periodically fetch this list (or it's published via CRDT). Verified wallets get `TrustLevel = 3`.
   - This can be a simple REST API initially: `GET /api/kyc/verified` → `["wallet1", "wallet2", ...]`
   - Signed by the KYC provider's key to prevent tampering.

**Files to change:**
- New package: `src/internal/kyc/` — client for KYC oracle
- `src/internal/protocol/node_table.go` — periodic trust level refresh

## Suggested Priority and Timeline

| Phase | Effort | Dependencies | Suggested Order |
|-------|--------|-------------|----------------|
| 1. Binary Signing | Low | None | First |
| 2. TLS Verification | Low | None | First (parallel with 1) |
| 3. Rate Limiting | Medium | None | Second |
| 4a. Wallet-Signed Identity | Medium | Wallet code exists | Third |
| 4b. Trust-Aware Routing | Medium | 4a | Fourth |
| 4c. Per-User Trust | Low | 4b | Fifth |
| 4d. KYC Integration | High | 4b, external KYC service | Last |

Phases 1 and 2 can ship independently and immediately. Phase 3 is self-contained. Phase 4 should be rolled out incrementally (a-d) with each sub-phase being independently shippable.
